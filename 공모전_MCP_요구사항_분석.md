## 2. 참가 파이프라인 (MCP를 "어떻게" 써야 하나)

```
① 로컬에서 MCP 서버 개발 (개발가이드 준수, MCP Inspector로 점검)
② PlayMCP in KC (playmcp.kakaocloud.io)에 배포
   - Git 소스 빌드(저장소 루트에 Dockerfile 필수) 또는 컨테이너 이미지
   - 무상 제공, 계정당 2대, 접수 기간(6/15~7/14)에만 발급
   - Active 되면 Endpoint URL 발급
③ PlayMCP 개발자 콘솔(playmcp.kakao.com/console)에 등록
   - Endpoint 입력 → "정보 불러오기" 성공해야 함
   - 먼저 "임시 등록" → 도구함 추가 → PlayMCP AI채팅으로 충분히 테스트
   - 완료되면 "심사 요청"
④ 심사 통과 → 공개 상태를 "나에게만 공개" → "전체 공개"로 전환 (안 하면 접수 제외)
⑤ 공모전 페이지 [Player 예선 참여] 비즈폼으로 접수 (MCP 상세페이지 URL 제출)
```

- ⚠️ 카카오클라우드 사용은 예선 필수 (지원 수량 소진 시 별도 공지로 예외 가능)
- ⚠️ 게시 후 Tool 정보 변경 시 "정보 불러오기" 재실행 = **재심사 발생**
- MCP 개발 이슈는 카카오가 도와주지 않음 (KC 오류만 지원)

## 3. 기술 스펙 (PlayMCP 서버 개발가이드, 2026-06-12 업데이트)

### 프로토콜
- MCP 스펙 버전: **2025-03-26 ~ 2025-11-25**
- **Streamable HTTP만** 지원 (stdio ❌, SSE ❌)
- **Remote 서버만** (공개 URL 필수) / **Stateless 권장** (no session)
- 인증 필요 시: 표준 OAuth 또는 커스텀 헤더
- 공식/활발한 SDK 사용 필수, MCP Inspector 사전 점검 필수
- **PlayMCP는 Tools만 지원 — Resource, Prompt 미지원**

### Tool 구성 규칙
| 항목 | 규칙 |
|---|---|
| 개수 | **3~10개 권장** (20개 초과 금지) |
| 이름 | 영숫자/`_`/`-`만, 1~128자, 중복 금지, "kakao" 포함 금지(대소문자 무관) |
| 필수 property | name, description, inputSchema, **annotations 5종 전부** (title, readOnlyHint, destructiveHint, openWorldHint, idempotentHint) |
| description | **영문 권장**, 서비스명 포함(영·국문 병기, 예: "...from Melon(멜론)"), 1,024자 이내 |
| result | 최소 크기, **24KB 초과 시 에러 = 반려**, raw API 응답 금지, 마크다운 정제 권장 |

### 성능 요구 ← 아이디어 설계를 직접 제약
- **툴 응답속도: 평균 100ms 이내, p99 3,000ms 필수**
- timeout 빈발, 과도한 리디렉션/크롤링 지연/불필요한 외부 호출 → 반려

### OAuth (사용자 인증 쓸 경우)
- Redirect URI: `https://playmcp.kakao.com/api/v1/applied-mcps/{mcpId}/authorize/oauth:callback`
- 개인정보 제3자 제공 동의 화면 권장 (양식 제공됨)
- 인증 만료 시 401 응답 필수

## 5. 본선: Kakao Tools

- 본선 진출작만 입점. **Widget 추가 스펙**으로 향상된 답변(시각적 UI) 제공
- PlayMCP보다 **더 엄격한 MCP 표준** 요구, 상세 스펙은 본선 진출 후 안내로 추정
- Kakao Tools 반영 시 tool name에 PlayMCP 지정 prefix 자동 부착 → tool 이름에 서비스명 넣을 필요 없음
- 본선 = **카톡 사용자 투표**가 반영 → 일반 사용자에게 매력적/시각적으로 어필하는 서비스가 유리

## 6. 새 아이디어 결정에 주는 제약 조건 (요약)

1. **속도가 왕**: 평균 100ms·p99 3s 기준 때문에 "서버 안에서 외부 API 여러 개 체이닝 + LLM 요약" 구조는 위험. **MCP는 정제된 데이터를 빠르게 반환하고, 요약·추론은 클라이언트 LLM에 맡기는 구조**가 정석. 느린 소스는 캐시/사전 적재 필요 (단, 제3자 데이터 DB 적재는 법적 이슈 — 기존 리서치 PART 1 참고)
2. **"LLM 웹검색으로 되는 것"은 탈락**: 아이디어 1차 필터. 고유 데이터·고유 연산·실시간성·개인화 중 하나는 있어야 함
3. **데이터 출처 증빙 가능해야 함**: 공식 API·공공데이터·자체 생성 데이터가 안전 (기존 크롤링 리스크 결론 재확인됨)
4. **Tool 3~10개, 응답 24KB 이내**로 설계 → 기능 욕심 금지, 좁고 깊게
5. **2개 출품 가능** → 본명 아이디어 + 가벼운 서브 아이디어 병행 전략 유효
6. **본선은 투표전** → 데모 한 방(위젯으로 보여줄 시각적 결과물)이 있는 아이디어가 유리
7. **Dockerfile로 배포** → 스택 자유 (Python FastMCP / TS SDK 아무거나), 단 Streamable HTTP + stateless
